Als **Anwendungssoftware** werden Computerprogramme bezeichnet, die genutzt werden, um eine nützliche, nicht systemtechnische Funktionalität zu unterstützen. Sie dienen der **Lösung von Benutzerproblemen**. Beispiele für Anwendungsgebiete sind: Bildbearbeitung, E-Mail-Programme, Webbrowser, … In information technology, an **application (app)** is a computer program **designed to help people perform an activity**.
Schlussfolgerungen
* Anwendungen besitzen eine Benutzeroberfläche * Apps sind kleine Anwendungen (App vs. Application) * Webanwendungen sind mit den Webtechnologien realisierte Anwendungen
Der Begriff **Dienst** (auch Service oder Daemon) beschreibt in der Informatik eine **technische, autarke Einheit**, die **zusammenhängende Funktionalitäten** zu einem Themenkomplex bündelt und über eine **klar definierte Schnittstelle** zur Verfügung stellt. Typische Beispiele sind hier Webservices, die Funktionalitäten für Dritte über das Internet verfügbar machen, Netzwerkdienste oder Systemdienste.
A service is a mechanism to enable access to one or more capabilities, where the access is provided using a prescribed interface, consistent with constraints and policies as specified by the service description.
Schlussfolgerungen
* Services besitzen in der Regel keine Benutzeroberfläche sondern laufen im Hintergrund * Ein Server stellt Services über das Netzwerk bzw. das Internet zur Verfügung * Services können aber auch auf andere Weise, beispielsweise als Betriebssystemfunktion realisiert werden * Webservices nutzen die Webtechnologien, um Services über das Internet verfügbar zu machen * Microservices kapseln größere Anwendungsfunktionen und bieten hierfür meist eine Webservice-Schnittstelle
* **Webanwendung:** Wird über HTTP aufgerufen und nutzt HTML/CSS/JavaScript zur Erzeugung der Benutzeroberfläche. * **Webservice:** Nutzt ebenfalls HTTP als Transportprotokoll, überträgt jedoch weitgehend nur Rohdaten darüber. * Moderne (clientseitige) Webanwendungen nutzen in der Regel Webservices Kommunikation mit dem Backend.

Im Gegensatz zu früher ist es heute üblich, dass viele Anwendungen und Apps auf Dienste im Internet zugreifen. Dies reicht vom einfachen Auslagern einzelner Funktionen bis zu Anwendungen, bei denen die komplette Anwendungslogik entfernt wurde und der verbleibende Teil lediglich die Benutzeroberfläche implementiert. * Clientseitige Webanwendungen * Mobile Applikationen * Desktop-Programme mit „Cloudfunktionen” * Smart TV Apps * … Im Falle von Webanwendungen ist es heute üblich, fachliche Logik und Oberfläche zu trennen, so dass derselbe Webservice auch für Mobile Clients und andere Benutzeroberflächen verwendet werden kann. Aber auch um B2B-Kontext oder zur Kommunikation mehrerer Anwendungen untereinander kommen REST-Webservices häufig zum Einsatz.
Ist ein Webservice dafür gemacht, von unterschiedlichen Anwendungen aufgerufen zu werden, spricht man heute auch von einer API. Gemeint ist damit ein **Application Programming Interface**, das seine Funktionen verschiedenen Anwendungen zur Verfügung stellt. Ursprünglich kommt dieser Begriff aus der Betriebssystementwicklung, wo er für die Systembibliotheken zur Nutzung der Betriebssystemfunktionen steht: * **WIN32-API:** Interne API von Microsoft Windows zur Erstellung von Windows-Anwendungen * **Java API:** Die mitgelieferte und sehr umfangreiche Klassenbibliothek der Programmiersprache Java * **Android API:** Aus der Java API abgeleitete Schnittstellen zur Programmierung von Android Apps In der objektorientierten Programmierung wird auch die öffentliche Schnittstelle einer Klasse gelegentlich als API bezeichnet. Im Rahmen des **API-driven Business** gehen heute immer Firmen dazu über, ihre Produkte und Dienstleistungen per **REST API** zur Verfügung zu stellen. Durch die Integration der APIs in fremde Webseiten, Anwendungen und Mobile Apps eröffnet sich auf diese Weise einen zusätzlicher Vertriebskanal.

    • Der Inhalt der Datei muss im Request Body mitgeschickt werden

    DELETE

    • Daten auf dem Server löschen
    • Die Anfrage darf keinen Request Body enthalten
    • Der Server antwortet mit einer Bestätigung oder einer Fehlberbeschreibung

    HTTP/0.9

    Dies war die ursprüngliche von Tim Berners-Lee entwickelte HTTP-Version, wie sie 1991 veröffentlicht wurde. In dieser Version gab es nur GET-Anfragen und noch keine Header Fields. Eine Anfrage bestand daher lediglich aus einer Zeile:

    GET /index.html

    Die Antwort des Servers beinhalte nur den Inhalt der abgerufenen Datei, sonst nichts.

    HTTP/1.0

    1996 wurde HTTP/1.0 veröffentlicht. Ab hier haben die HTTP-Nachrichten den oben gezeigten Aufbau.

    HTTP/1.1

    Diese Version wurde 1999 veröffentlicht und brachte weitreichende Verbesserungen. Unter anderem die Möglichkeit, die Verbindung zwischen zwei Anfragen offen zu halten. Denn zuvor wurde die Verbindung nach jeder Antwort automatisch getrennt. Da aber gerade der Verbindungsaufbau bei TCP/IP sehr langsam ist, führte dies zu langen Wartezeiten, wenn mehrere Dateien vom selben Server abgerufen werden mussten.

    HTTP/2

    2015 wurde HTTP/2 offiziell verabschiedet. In seinen Grundzügen basiert es auf dem von Google entwickelten SPDY-Protokoll. Die Inhalte der HTTP-Nachrichten sind weitgehend gleich geblieben, jedoch werden sie nicht mehr in der oben gezeigten Textform kodiert. Stattdessen kommt ein kompaktes und in vielen Dingen eindeutiger spezifiziertes Binärformat zum Einsatz.¹

    Als weitere größere Neuerung führt HTTP/2 Server Push ein, wodurch der Server beim Abruf einer HTML-Seite selbstständig weitere Dateien an den Browser schicken kann. Somit können Stylesheets, Skripte und Bilder schon an den Browser geschickt werden, bevor dieser überhaupt weiß, dass er sie benötigen wird.

    Stand 2020 hat sich HTTP/2 noch nicht vollständig gegen HTTP/1 durchgesetzt. Sehr viele Webseiten verwenden weiterhin HTTP/1. Inzwischen unterstützen aber alle Server, Browser und Frameworks das neue Protokoll.

    ¹ Beispielsweise war bei HTTP/1.x nie definiert, welche Bytefolge ein Zeilenende markiert.

    Angeleitet vom Konzept des **entfernten Methodenaufrufs** entwickelte sich Anfang der 2000er-Jahre **SOAP** zu einem weit verbreiteten Standard. Die Abkürzung stand dabei ursprünglich für **Simple Object Access Protocol*, wobei das Protokoll weder simpel noch objekt-orientiert ist. Denn anstelle von Objekten definiert ein Webservice hier eine Reihe von Funktionen, die mit einer **HTTP/POST-Anfrage** entfernt aufgerufen werden können. Anfrage und Antwort werden dabei im **XML-Format** kodiert und enthalten neben dem Namen der gerufenen Funktion auch alle Anfrageparameter und Rückgabewerte. Heute nimmt die Bedeutung von SOAP immer mehr ab, so dass SOAP in vielen Fällen durch REST verdrängt wurde. Die Grundidee eines Webservice-Aufrufs lässt sich daran aber immer noch gut erklären.
    Im Gegensatz zu SOAP orientiert sich REST vollständig an den Möglichkeiten von HTTP und überträgt diese konzeptionell vom Abruf einer Webseite auf die **Verwaltung beliebiger „Ressourcen”**, wobei Ressourcen in diesem Zusammenhang nahezu alle Dinge sein können, deren Inhalt in einer geeigneten Form zwischen Client und Server ausgetauscht werden kann. Oft handelt es sich deshalb um Einträge in einer Datenbank. Aber auch IoT-Anwendungsfälle, bei denen jede Ressource für ein dingliches Objekt der realen Welt steht, sind denkbar. Jede Ressource besitzt hierfür eine eindeutige URL, die mit den üblichen HTTP-Methoden (GET, PUT, POST, DELETE, …) angesprochen wird. Die Abkürzung REST steht dabei für **Representational State Transfer**, was wie folgt zu lesen ist: * **Representational:** Grundsätzlich ist REST nicht auf ein bestimmtes Datenformat festgelegt. Am häufigsten werden die Daten zwar im JSON-Format übertragen. Viele Server unterstützen darüber hinaus aber noch weitere Formate wie XML, zwischen denen der Client wählen kann. * **State Transfer:** Von der Nutzung einer Datenbank abgesehen ist die Serverseite eines REST-Webservices immer zustandslos. Auf Serverseite können daher keine Daten im Hauptspeicher temporär abgelegt werden, um diese bei einer Folgeanfrage weiterzubearbeiten. Temporäre Daten müssen entweder direkt in einer angeschlossenen Datenbank persistiert oder vom Client bei jeder Anfrage mitgesendet werden. Dies hat u.a. auch zur Folge, dass fast jede Anfrage bzw. Antwort den kompletten Inhalt einer Ressource beinhaltet. Bei den URLs wird zwischen Ressourcen, Collections, Stores und Actions unterschieden.

    Ein einzelner Datensatz wird Ressource genannt. Jede Ressource besitzt eine eindeutige URL, unter der sie aufgerufen werden kann.

    https://example.com/api/{resource}/{id}
    https://example.com/api/customers/47110815

    { "id": "47110815", "first_name": "Martina", "last_name": "Musterfrau", "registration_date": "2023-02-09" }
    GET PUT POST PATCH DELETE
    Ressource lesen Ressource überschreiben ––– Einzelne Werte ändern Ressource löschen

    Mehrere Datensätze werden in einer Collection gesammelt, ähnlich wie die Tabellen einer SQL-Datenbank.

    https://example.com/api/{collection}
    https://example.com/api/customers

    [ {"id": "47110815", "name": "Martina Musterfrau"}, {"id": "47110816", "name": "Toll Es Beispiel"}, {"id": "47110817", "name": "Klaus Kunde"} ]
    GET PUT POST PATCH DELETE
    Alle Ressourcen Lesen ––– Neue Ressource anlegen ––– Alle Ressourcen löschen

    Im Gegensatz zu Collections speichern Stores zwar ebenfalls mehrere Datensätze, die URL wird jedoch vom Client festgelegt bzw. ist fest vorgegeben und daher vorab bekannt.

    https://example.com/api/{collection}/{id}/{store}
    https://example.com/api/customers/47110815/favorites

    [ {"article": "9573387", "date_added": "2023-01-02"}, {"article": "5975741", "date_added": "2023-01-14"}, {"article": "1572246", "date_added": "2023-02-07"} ]
    GET PUT POST PATCH DELETE
    Inhalte auslesen Einträg hinzufügen ––– Einzelne Werte ändern Eintrag löschen

    Prozedurale Aktionen, die nicht dem üblichen Lesen und Schreiben entsprechen, werden Controller¹ oder auch Actions genannt.

    https://example.com/api/{collection}/{id}/{action}
    https://example.com/api/customers/47110815/checkout

    ¹ Der Name Controller ist unglücklich gewählt, weil die serverseitigen Klassen zur Implementierung eines REST-Webservices in den meisten Frameworks auch Controller genannt werden. In der Vorlesung verwenden wir daher lieber den Begriff Action.

    Jedes HTTP-Verb und jeder Stautscode besitzt in REST eine genau definierte Semantik. Bei einer Anfrage muss daher immer das passende HTTP-Verb gesendet werden. Ebenso muss der empfangene Statuscode ausgewertet werden, um den Erfolg der Anfrage zu prüfen.

    HTTP-Verben
    GET Daten lesen
    POST Daten unter einer vom Server generierten URL speichern
    PUT Daten unter einer bereits bekannten URL speichern
    PATCH Einzelne Felder eines vorhandenen Datensatzes ändern
    DELETE Daten löschen
    POST/PUT Aufruf einer prozeduralen Aktion

    GET Liste abrufen

    GET-Anfragen an einer Collection oder einen Store führen in der Regel dazu, dass eine Zusammenfassung aller enthaltenen Daten gesendet wird. Oftmals können Anzahl, Umfang und Sortierung der Daten über URL-Parameter gesteuert werden.

    HTTP-Anfrage

    GET /api/xfiles2018?sort_by=episode HTTP/1.1 Host: example.com Accept: application/json

    HTTP-Antwort

    HTTP/1.1 200 Ok Content-Type: application/json; charset=utf-8 [{ "episode": "2AYW01", "title": "My Struggle III" }, { "episode": "2AYW02", "title": "This" }, { "episode": "2AYW03", "title": "Plus One" }, { "episode": "2AYW04", "title": "The Lost Art of Forehead Sweat" }, { "episode": "2AYW05", "title": "Ghouli" }, { "episode": "2AYW06", "title": "Kitten" }, { "episode": "2AYW07", "title": "Rm9sbG93ZXJz" }, { "episode": "2AYW08", "title": "Familiar" }, { "episode": "2AYW09", "title": "Nothing Lasts Forever" }, { "episode": "2AYW10", "title": "My Struggle IV" }]
    GET Ressource abrufen

    GET-Anfragen an eine einzelne Ressourcen führen dazu, dass der komplette Datensatz übermittelt wird. Manche Webservices erlauben auch hier, über URL-Parameter nur bestimmte Felder anzufordern. Die meisten Webservices liefern aber einfach die komplette Ressource.

    HTTP-Anfrage

    GET /api/xfiles2018/2AYW04 HTTP/1.1 Host: example.com Accept: application/json

    HTTP-Antwort

    HTTP/1.1 200 Ok Content-Type: application/json; charset=utf-8 { "episode": "2AYW04", "title": "The Lost Art of Forehead Sweat", "directed_by": "Darin Morgan", "written_by": "Darin Morgan", "usa_air_date": "January 24, 2018", "de_air_date": "March 15, 2018", "running_time": 43, "guest_starts": [ "Brian Huskey", "Stuart Margolin" ] }
    POST Ressource anlegen

    POST-Anfragen werden in der Regel an eine Collection geschickt, um neue Daten anzulegen. Der Server vergibt daraufhin eine neue URL und teilt diese zusammen mit einer Kopie der gespeicherten Daten mit. Stellt der Server fest, dass der anzulegende Datensatz bereits vorhanden ist, kann er diesen entweder überschreiben oder dies mit dem Statuscode 409 Conflict ablehnen.

    HTTP-Anfrage

    POST /api/xfiles2018/2AYW04/reviews HTTP/1.1 Host: example.com Content-Type: application/json; charset=utf-8 Accept: application/json { "star_rating": 10, "text": "One of the funniest episodes …" }

    HTTP-Antwort

    HTTP/1.1 201 Created Location: /api/xfiles2018/2AYW04/reviews/42 Content-Type: application/json; charset=utf-8 Cache-Control: no-cache { "id": 42, "episode": "2AYW04", "star_rating": 10, "text": "One of the funniest episodes …" }
    PUT Ressource aktualisieren

    Bei einer PUT-Anfrage kennt der Client die URL der abzuspeichernden Daten bereits, so dass die Anfrage direkt an die Ressource geschickt werden kann. Meistens wird dies zum Überschreiben vorhandener Datensätze genutzt. Es gibt aber auch Services, welche eine Neuanlage auf diese Weise erlauben. Manche Webservices liefern auch hier als Antwort eine Kopie der gespeicherten Daten. Üblich ist aber auch, einfach mit Status 204 No Content zu antworten und davon auszugehen, dass der Client due vollständigen Daten entweder bereits kennt oder mit einer nachfolgenden GET-Anfrage abrufen kann.

    HTTP-Anfrage

    PUT /api/xfiles2018/2AYW07 HTTP/1.1 Host: example.com Content-Type: application/json; charset=utf-8 Accept: application/json { "episode": "2AYW07", "title": "Rm9sbG93ZXJz", "directed_by": "Glen Morgan", "written_by": "Shannon Hamblin & Kristen Cloke", "usa_air_date": "February 28, 2018", "de_air_date": "April 4, 2018", "running_time": 43, "guest_starts": [ "Candus Churchill" ] }

    HTTP-Antwort

    HTTP/1.1 204 No Content Location: /api/xfiles2018/2AYW07
    PATCH Einzelne Felder aktualisieren

    Soll ein vorhandener Datensatz aktualisiert werden, kann häufig eine PATCH-Anfrage gesendet werden, in welcher nur die zu überschreibenden Felder enthalten sind. Auf diese Weise wird unnötiger Traffic zwischen Client und Server vermieden.

    HTTP-Anfrage

    PATCH /api/xfiles2018/2AYW09 HTTP/1.1 Host: example.com Content-Type: application/json; charset=utf-8 Accept: application/json { "directed_by": "James Wong", "written_by": "Karen Nielsen" }

    HTTP-Antwort

    HTTP/1.1 200 Ok Location: /api/xfiles2018/2AYW09 Content-Type: application/json; charset=utf-8 Cache-Control: no-cache { "episode": "2AYW09", "title": "Nothing Lasts Forever", "directed_by": "James Wong", "written_by": "Karen Nielsen", "usa_air_date": "March 14, 2018", "de_air_date": "April 11, 2018", "running_time": 44, "guest_starts": [ "Fiona Vroom", "Carlena Britch", "Jere Burns" ] }
    DELETE Ressource löschen

    Eine DELETE-Anfrage dient, wie der Name bereits sagt, dem Löschen von Daten. Sie wird daher in der Regel an eine einzelne Ressource geschickt, kann aber auch an eine Collection oder einen Store geschickt werden, um alle Daten auf einmal zu löschen. Als Antwort wird häufig nur eine Bestätigung ohne Dateninhalt geliefert.

    HTTP-Anfrage

    DELETE /api/reviews/13 HTTP/1.1 Host: example.com

    HTTP-Antwort

    HTTP/1.1 204 No Content
    PUT Aktion ausführen

    Prozedurale Aktionen werden durch eine PUT- oder POST-Anfrage ausgelöst, keinesfalls durch GET!¹ Viele Webservices verwenden POST, weil dies auch bei SOAP-Webservices (dem zweiten großen Standard neben REST) praktiziert wird, PUT wäre aber eigentlich logischer, da der Client die aufzurufende URL ja genau kennt.

    ¹ GET ist nur zum Abruf bereits vorhandener Daten gedacht und darf deshalb keine Seiteneffekte haben.

    HTTP-Anfrage

    PUT /api/xfiles2018/2AYW01/stream HTTP/1.1 Host: example.com Content-Type: application/json; charset=utf-8 Accept: application/json { "to_device": "smart_tv1", "start_at_timecode": 0, "audio": "english" }

    HTTP-Antwort

    HTTP/1.1 200 Ok Content-Type: application/json; charset=utf-8 { "stream_id": "2AYW01.smart_tv1.english" }

    Bildnachweis: Geek & Poke: 30.11.2009

    Entsprechend den Empfehlungen von Roy Fielding, dem Erfinder von REST, verhalten sich einige REST-Webservices **hypermedia driven**, indem Sie das **HATEOAS-Prinzip** (Hypertext is the engine of the application state) berücksichtigen. Dieses sieht vor, dass ein entfernter Client die Struktur eines Webservices durch Verlinkungen innerhalb der Antworten komplett dynamisch entdecken kann. Jede Antwort beinhaltet hierfür neben den eigentlichen Nutzdaten zusätzlich eine Liste mit Hyperlinks für alle zulässigen Folgeaktionen. Zur Verdeutlichung soll hier eine einfache Onlinebanking-Anwendung betrachtet werden. Über die Verlinkungen kann das Frontend erkennen, ob eine Konto beispielsweise ausreichend Deckung zum Tätigen von Überweisungen besitzt. Ohne Verlinkungen muss der Client „wissen“, wann welche Aktion zulässig ist.

    Es werden zwar die richtigen Daten zu einem Konto geliefert. Das Backend teilt aber nicht mit, wann welche Aktion erlaubt ist.

    [ { "number": 3699623, "type": "giro" }, { "number": 1013699623, "type": "savings" } ] { "account_number": 3699623, "bank_name": "Parkbank", "currency": "EUR", "amount": "2411.83", "state": "covered" }

    Durch Verlinkungen innerhalb der REST-Antworten teilt das Backend mit, wann welche Aktion erlaubt ist und wie diese aufzurufen ist.

    Während SOAP-Webservices von Anfang an mit der **Webservice Definition Language** (meist einfach nur WSDL abegkürzt) in einem maschinenlesbaren formal beschrieben wurden, gab es für REST-Webservices lange Zeit keinen vergleichbaren Standard. Dabei bietet eine formale Beschreibung oftmals folgende Möglichkeiten: * Generieren von Client-Code zum Aufruf des Webservices * Generieren von Server-Code zur Implementierung des Webservices * Automatische Validierung von Anfragen und Antworten * Generieren statische Dokumentationsseiten * … Als möglichst leichtgewichtiger Ansatz kamen REST-Webservices lange ohne eine formale Beschreibung und entsprechende Werkzeuge aus. Meistens wurden sie einfach nur in Prosatext auf einer Begleitwebseite beschrieben, sofern es sich um öffentliche APIs handelt (vgl. z.B. die Wikipedia API). Inzwischen hat sich jedoch das **OpenAPI-Format** (vormals Swagger genannt) als De-Facto-Standard durchgesetzt.
    Generell sind zwei Vorgehensweisen bei der Programmierung denkbar: * **Code First:** Die API-Spezifikation wird anhand der Webservice-Implementierung automatisch generiert (z.B. mit dem Node.js-Paket express-oas-generator). * **Contract First:** Die Spezifikation wird unabhängig von der konkreten Implementierung erstellt und der Servercode anschließend entweder daraus generiert oder von Hand implementiert. REST-Webservices werden allerdings fast immer **Contract First** erstellt. Die hierfür notwendigen OpenAPI-Dokumente werdem im JSON- oder YAML-Format erstellt, wobei YAML meist bevorzugt wird. Gute Anleitungen finden sich u.a. hier: * Swagger OpenAPI Guide * Smartbear OpenAPI Tutorial

    Die Startadresse des Webservices soll eine Übersicht mit allen Collections liefern, wobei zu jeder Collection die URLs, HTTP-Verben und URL-Parameter zum Durchsuchen der Collections und zur Anlage neuer Einträge mitgeteilt werden sollen.

    HATEOAS_Collection Beschreibung einer einzelnen Collection
    Feld Datentyp Beschreibung
    _name String Name der Collection
    query HATEOAS_Link Link zum Durchsuchen der Collection
    create HATEOAS_Link Link zur Anlage neuer Einträge
    HATEOAS_Link Details zu einem Link
    Feld Datentyp Beschreibung
    url String URL-Pfad relativ zur Startadresse des Webservices
    method String Zu verwendende HTTP-Methode (z.B. GET oder POST)
    query_params Array[String]

    Namen aller erlaubten URL-Parameter

    Der Einfachheit halber ohne weitere Details, wie z.B. dem Datentyp der Parameter oder welche Parameter optional sind. Es wird davon ausgeganen, dass es keine Pflichtparameter gibt und der Aufrufer weiß, in welchem Format die Werte übergeben werden müssen.

    [ { "_name": "address", "query": { "url": "/address", "method": "GET", "query_params": ["search", "first_name", "last_name", "phone", "email"] }, "create": { "url": "/address", "method": "POST", } } ]

    Jede Ressource des Webserivces (hier also jeder Adressbucheintrag) soll darüber hinaus alle notwendigen Informationen liefern, damit ein Client weiß, wie er die Ressource lesen, ändern und löschen kann.

    HATEOAS_Resource Links zu einer einzelnen Ressource
    Feld Datentyp Beschreibung
    read HATEOAS_Link Link zum Lesen der Ressource
    update HATEOAS_Link Link zum Überschreiben der Ressource
    patch HATEOAS_Link Link zum Ändern einzelner Felder der Ressource
    delete HATEOAS_Link Link zum Löschen der Ressource
